# i18n Handoff: Remaining Work

This document is a complete handoff for finishing the Abzar i18n implementation. It assumes you have zero prior context. Read this entire document before starting any work.

---

## What's Already Done

The i18n infrastructure is fully operational. Two locales are supported: English (`en`) and Persian (`fa`). All URLs require a locale prefix (`/en/...`, `/fa/...`). The root `/` redirects to `/en`. RTL layout works for Persian. Fonts load per-locale. The locale switcher is in the header. The sitemap includes hreflang alternates. PWA manifests exist per locale.

**Fully translated to Persian:**
- All navigation/UI chrome: sidebar, header, breadcrumbs, badges ("Coming Soon", "Beta", "New"), search placeholder, 404 page, tool page section headings ("About this tool", "How to use", "Related Tools")
- All 14 category names and descriptions
- Homepage section headings ("Categories", "Popular Tools", etc.)
- Locale switcher labels

**Not translated — still English everywhere:**
- 200+ tool names and descriptions (`messages/fa/tools.json` — English placeholders)
- 30+ tool about/howTo content blocks (`messages/fa/tool-content.json` — English placeholders)
- All strings inside tool components (buttons, labels, placeholders, toasts — hardcoded in TSX)
- Shared UI primitives (CopyButton, DownloadButton, FileDropZone, InputOutputLayout)
- Site tagline/subtitle on homepage
- Homepage quick-filter pills
- Search results (shows English names even on `/fa`)
- JSON-LD structured data (`inLanguage` hardcoded to `"en"`)
- OG images (don't exist for either locale)
- Service worker precache patterns (doesn't account for locale paths)

See `docs/I18N_CHANGELOG.md` for the full list of files changed during the initial implementation.

## Remaining Work Items

### 1. Translate `messages/fa/tools.json` (human translation)

**What:** This file at `messages/fa/tools.json` contains every tool's `name` and `description` — currently all in English. A Persian translator needs to translate every entry.

**Structure:**
```json
{
  "tools": {
    "word-counter": {
      "name": "Word & Character Counter",        // ← translate this
      "description": "Count words, characters..." // ← translate this
    },
    // ... 200+ entries
  }
}
```

**Scale:** ~200 entries, each with a name (2-5 words) and description (one sentence).

**Where it's used:** Tool names/descriptions appear on the homepage (featured tools), category pages (tool cards), tool pages (title + meta description), search results, breadcrumbs (tool name), sidebar could use them in future, and `<title>` tags.

**How to test:** Visit `/fa` and `/fa/tools/text/word-counter` — tool names and descriptions should appear in Persian.

**Important:** The English version of this file (`messages/en/tools.json`) is auto-generated by `scripts/generate-messages.ts` from `lib/tools-registry.ts` and is gitignored. The Persian file is committed and hand-maintained. When new tools are added to the registry, someone must also add Persian translations to `messages/fa/tools.json` manually.

**Structure:**
```json
{
  "toolContent": {
    "json-formatter": {
      "about": "Paste or type JSON to instantly format...",  // ← translate
      "howTo": [                                              // ← translate each

        "Click Format to pretty-print, or Minify to compress",
        "Use the tree view to explore nested structures",
        "Click Copy to copy the result to your clipboard"
```

**Scale:** ~30 tool entries, each with an `about` paragraph and 3-5 `howTo` steps.

**Where it's used:** Below each tool's interactive widget on the tool page. Rendered by `components/layout/tool-page.tsx`. The page uses `tContent.has('slug.about')` to check existence before rendering, so missing entries just hide the section.

**How to test:** Visit `/fa/tools/developer/json-formatter` and scroll below the tool widget — the about/howTo sections should be in Persian.

**Same maintenance rule as tools.json:** English version is auto-generated and gitignored; Persian is committed and hand-maintained.

### 3. Translate strings inside tool components (per-tool i18n)

This is the largest piece of work. Every tool component has hardcoded English strings in JSX.

**Scale:** 74 tool component files, most with 5-15+ hardcoded strings each. Estimated 400-600 strings total across all tools.

**What kinds of strings:**

| Type | Examples | Count (approx) |
|------|----------|-----------------|
| Button labels | "Copy", "Clear", "Reset", "Generate", "Format", "Minify", "Convert", "Download", "Save", "Delete", "Add", "Remove", "Play", "Stop", "Pause", "Export", "Import" | ~45 |
| Input placeholders | "Paste your JSON here...", "Type or paste text...", "Enter a URL...", "New habit name..." | ~40 |
| Section headings / labels | "Input", "Output", "Words", "Characters", "Sentences", "Length", "This Week", "Generated N passwords" | ~30 |
| Toast messages | "Copied to clipboard!", "Failed to copy" (via CopyButton), various tool-specific toasts | ~10 |
| Empty states | "Add a habit to start tracking.", "No results found" | ~10 |
| Aria-labels | "Delete", "Remove habit", "Decrease", "Increase", "Swap colors" | ~15 |
| Table headers | "Habit", "Streak", "Name", "Value", "Type" | ~10 |
| Inline text | Readability score labels ("Flesch-Kincaid", "Gunning Fog"), unit names, descriptions | many |

**Architecture for per-tool translations:**

The infrastructure is partially prepared. In `app/[locale]/tools/[category]/[tool-slug]/page.tsx` at line 273, there's this comment:

```ts
// Per-tool messages (messages/{locale}/tool/{slug}.json) can be merged here
// once individual tool translation files are created.
```

And the tool page already wraps children in `<NextIntlClientProvider messages={mergedMessages}>`, so tool components can use `useTranslations()`.

**Recommended approach:**

1. Create per-tool message files at `messages/en/tool/{slug}.json` and `messages/fa/tool/{slug}.json`. Example for `json-formatter`:

```json
// messages/en/tool/json-formatter.json
{
  "jsonFormatter": {
    "format": "Format",
    "minify": "Minify",
    "validate": "Validate",
    "clear": "Clear",
    "inputPlaceholder": "Paste your JSON here... e.g. {\"key\": \"value\"}",
    "outputPlaceholder": "Formatted output will appear here",
    "validJson": "Valid JSON",
    "invalidJson": "Invalid JSON"
  }
}
```

2. Update `i18n/request.ts` to dynamically load per-tool messages. Since all messages are merged into one object and the tool slug isn't known at request config time, the simplest approach is to load all per-tool files. But that defeats code-splitting. Instead, modify the tool page (`app/[locale]/tools/[category]/[tool-slug]/page.tsx`) to:

```ts
// In the page component, after line 275:
let perToolMessages = {};
try {
  perToolMessages = (await import(`@/messages/${locale}/tool/${slug}.json`)).default;
} catch {
  // No per-tool translations yet — that's fine
}
const mergedMessages = { ...(await getMessages()), ...perToolMessages };
```

**Important:** This dynamic import approach won't work with `@/` alias in Next.js static builds. You'll need to use relative paths:
```ts
perToolMessages = (await import(`../../../../messages/${locale}/tool/${slug}.json`)).default;
```
Or, import all per-tool files in `i18n/request.ts` using a glob pattern if available, or enumerate them explicitly.

3. In each tool component, add `useTranslations('jsonFormatter')` (or whatever namespace) and replace hardcoded strings.

4. To auto-generate English per-tool message files, extend `scripts/generate-messages.ts` to extract strings from tool components. This is hard to automate — it's more practical to create them manually as each tool is translated.

**Recommended order:** Start with the 10 Phase 1 tools (most popular), then Phase 2, etc. The order from `ABZAR_SPEC.md`:

Phase 1: `json-formatter`, `base64`, `color-picker`, `password-generator`, `uuid-generator`, `word-counter`, `unit-converter`, `timestamp-converter`, `hash-generator`, `qr-generator`

Phase 2: `regex-tester`, `image-resizer`, `gradient-generator`, `markdown-editor`, `mortgage-calculator`, `pomodoro`, `subnet-calculator`, `jwt-decoder`, `cron-builder`, `contrast-checker`

### 5. Make search locale-aware

**Current state:** `lib/search.ts` creates a Fuse.js index from `tools-registry.ts` which has English-only `name`, `description`, and `tags`. Persian users typing Persian text in the search box will get zero results.

**File:** `lib/search.ts`
```ts
fuse = new Fuse(getAllTools(), {
  keys: [
    { name: "name", weight: 0.5 },
    { name: "description", weight: 0.3 },
    { name: "tags", weight: 0.2 },
  ],
  threshold: 0.4,
  ignoreLocation: true,
  minMatchCharLength: 1,
});
```

**Problem:** The Fuse index is a module-level singleton built from the English registry. There's no locale parameter.

**Solution approach:**

Option A — **Dual-index** (recommended): Build two Fuse indexes, one per locale. The English index uses tool registry data. The Persian index merges registry data with translated names/descriptions from `messages/fa/tools.json`.

```ts
import en_tools from '@/messages/en/tools.json';
import fa_tools from '@/messages/fa/tools.json';

const indexes: Record<string, Fuse<Tool>> = {};

function ensureIndex(locale: string) {
  if (!indexes[locale]) {
    const translations = locale === 'fa' ? fa_tools.tools : en_tools.tools;
    const items = getAllTools().map(tool => ({
      ...tool,
      localName: translations[tool.slug]?.name ?? tool.name,
      localDesc: translations[tool.slug]?.description ?? tool.description,
    }));
    indexes[locale] = new Fuse(items, {
      keys: [
        { name: "localName", weight: 0.4 },
        { name: "name", weight: 0.15 },       // still search English as fallback
        { name: "localDesc", weight: 0.25 },
        { name: "tags", weight: 0.2 },
      ],
      threshold: 0.4,
      ignoreLocation: true,
      minMatchCharLength: 1,
    });
  }
}

export function searchTools(query: string, locale: string = 'en'): Tool[] {
  const q = query.trim();
  if (!q) return [];
  ensureIndex(locale);
  return indexes[locale]!.search(q).map(r => r.item);
}
```

**Callers to update:**

1. `components/home-search.tsx` — line 31: `setResults(searchTools(value).slice(0, 8))` needs locale:
   ```ts
   const locale = useLocale();
   // ...
   setResults(searchTools(value, locale).slice(0, 8));
   ```

2. `components/layout/header.tsx` — has a similar search in the header command palette. Find the `searchTools` call there and add `locale`.

**Also:** The search results in `components/home-search.tsx` currently display `tool.name` and `tool.description` (English from registry) at lines 70-73. These should display translated names:
```tsx
// Currently:
<span className="font-medium text-sm">{tool.name}</span>
<p className="text-xs text-muted-foreground truncate">{tool.description}</p>

// Should be:
const tTools = useTranslations("tools");
// ...
<span className="font-medium text-sm">{tTools(`${tool.slug}.name`)}</span>
<p className="text-xs text-muted-foreground truncate">{tTools(`${tool.slug}.description`)}</p>
```

Same issue exists in the header search.

### 8. Localize JSON-LD structured data

**File:** `lib/json-ld.ts`

**Current issues:**
- Line 17: `browserRequirements: "Requires JavaScript. Works in Chrome, Firefox, Safari, Edge."` — hardcoded English
- Line 28: `inLanguage: "en"` — hardcoded, should be the current locale
- Line 14: `url` doesn't include locale prefix — should be `/${locale}/tools/${tool.category}/${tool.slug}`

**Solution:** Add `locale` parameter to `generateToolJsonLd`:

```ts
export function generateToolJsonLd(
  tool: Tool,
  locale: string,
  overrides?: { name: string; description: string }
) {
  return {
    "@context": "https://schema.org",
    "@type": "WebApplication",
    name: overrides?.name ?? tool.name,
    description: overrides?.description ?? tool.description,
    url: `${siteConfig.url}/${locale}/tools/${tool.category}/${tool.slug}`,
    applicationCategory: "UtilityApplication",
    operatingSystem: "Any",
    browserRequirements: locale === "fa"
      ? "نیاز به جاوااسکریپت دارد. در Chrome، Firefox، Safari و Edge کار می‌کند."
      : "Requires JavaScript. Works in Chrome, Firefox, Safari, Edge.",
    offers: { "@type": "Offer", price: "0", priceCurrency: "USD" },
    creator: {
      "@type": "Organization",
      name: siteConfig.name,
      url: siteConfig.url,
    },
    inLanguage: locale,
    isAccessibleForFree: true,
    featureList: tool.tags.join(", "),
  };
}
```

**Caller update:** `app/[locale]/tools/[category]/[tool-slug]/page.tsx` `generateMetadata` — pass `locale` to `generateToolJsonLd(tool, locale, { name: toolName, description: toolDesc })`.

### 10. Fix service worker precache patterns

**File:** `scripts/generate-sw.mjs`

**Current precache patterns (line 23-28):**
```js
globPatterns: [
  "index.html",
  "offline.html",
  "tools/*.html",
  "_next/static/chunks/*.css",
  "_next/static/*/_buildManifest.js",
  "_next/static/*/_ssgManifest.js",
  "_next/static/media/*.woff2",
],
```

**Problem:** With i18n, the static export structure is `out/en/index.html`, `out/fa/index.html`, `out/en/tools/text/word-counter.html`, etc. The patterns `index.html` and `tools/*.html` won't match the locale-prefixed paths.

**Fix:**
```js
globPatterns: [
  "en/index.html",
  "fa/index.html",
  "en/tools/**/*.html",
  "fa/tools/**/*.html",
  "_next/static/chunks/*.css",
  "_next/static/*/_buildManifest.js",
  "_next/static/*/_ssgManifest.js",
  "_next/static/media/*.woff2",
],
```

Or more generically:
```js
globPatterns: [
  "{en,fa}/index.html",
  "{en,fa}/tools/**/*.html",
  // ...
],
```

Also check if `offline.html` still exists at the right path.

**Also consider:** The service worker should serve the correct locale's offline page. This may require locale detection in the SW fetch handler (`sw/index.ts`).

## Adding a New Locale

If a third locale is added (e.g. Arabic `ar`):

1. Add `'ar'` to the `locales` array in `i18n/routing.ts`
2. Add `'ar'` to `rtlLocales` in `app/[locale]/layout.tsx` (if RTL)
3. Create `messages/ar/` with all 5 message files (copy from `fa/` as starting point)
4. Create `public/manifest.ar.json`
5. Arabic fonts may need adding in `app/[locale]/layout.tsx` and `app/globals.css`
6. Service worker patterns need updating
7. All `generateStaticParams` cross-products automatically pick it up
8. Sitemap generation automatically picks it up
